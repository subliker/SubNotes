# SubNotes — план проекта

## Что это

Лёгкий **компаньон поверх системного Календаря macOS**. Не дублирует календарь:
Google-события уже синкаются в Calendar.app через «Учётные записи интернета», а
SubNotes добавляет сверху то, чего в стандартном Календаре нет.

Три поверхности ценности:
1. **Список событий в поповере из menu bar** — клик по иконке открывает список
   ближайших событий (как виджет на Android, но из строки меню). Заменяет
   WidgetKit-виджет, чтобы уйти от подписи/App Group.
2. **Бегущая строка в menu bar** (а-ля RunCat) — напоминания умным появлением.
3. **Интерактивные оверлеи** поверх экрана — кастомизируемые скины (от карточки
   до «девочки-аниме с плакатом» или пролетающего самолёта с баннером).

Собственное окно приложения — **только настройки**.

> **Цвет как ключ кастомизации (сквозная идея).** В будущем индивидуальные
> настройки — правила уведомлений, шаблоны строк о событиях в тикере, выбор
> скина оверлея — привязываются к **цвету события**. Цвет выступает *ключом*
> группировки («все «оранжевые» события показывать скином X и за 30 мин»).
> Поэтому цвет — не декорация, а первоклассный идентификатор: архитектуру
> строим вокруг сменного «источника цвета» (`ColorKey`), см. раздел ниже.

> **Почему не WidgetKit-виджет:** настоящий desktop-виджет требует App Group для
> обмена данными с отдельным sandboxed-процессом, а App Group на macOS требует
> Team ID и provisioning — то есть какой-либо учётки разработчика. Это
> несовместимо с «полностью неподписанным» (проверено в Фазе 0). Поэтому список
> событий живёт в поповере главного процесса — ноль подписи, работает у всех.

## Ключевые решения

| Тема | Решение |
|---|---|
| Платформа | Нативно: **Swift + SwiftUI + AppKit** |
| Данные календаря | **EventKit** (чтение системных календарей). Google уже подключён через Internet Accounts — OAuth не нужен. |
| Режим данных | **Read-only** в MVP. Edit — возможно позже; границу read/write держим чистой, но edit-кода сейчас НЕ пишем. |
| Присутствие | **Только menu bar (LSUIElement)**, без Dock-иконки. |
| Окно приложения | Только экран **настроек**. |
| Открытие события | Deep-link в системный **Calendar.app**. |
| Напоминания | По **EKAlarm** событий. |
| Стиль | **Liquid Glass** → целимся в свежую macOS (26 Tahoe+), без back-compat компромиссов. |
| Распространение | **Open source, неподписанный `.dmg`**. Никаких платных Apple-аккаунтов. Раздача через GitHub Releases. |

## Архитектура

```
SubNotes/
├── Sources/App/          # menu-bar приложение (LSUIElement), один процесс
│   ├── SubNotesApp       # MenuBarExtra + поповер со списком событий
│   ├── TickerStatusItem  # NSStatusItem с бегущим текстом (RunCat-style)  [Фаза 3]
│   ├── OverlayEngine     # прозрачное NSWindow + движок скинов            [Фаза 4]
│   └── Settings          # окно настроек (SwiftUI)                        [Фаза 5]
└── Packages/CalendarCore # Swift package: модели + чтение EventKit + парсинг видеоссылок
```

**Поток данных:** EventKit → CalendarCore (нормализует) → прямо в UI того же
процесса (поповер, ticker, оверлеи). App Group/снапшот больше не нужны — всё в
одном процессе.

## Цветовая модель и кастомизация по цвету

Цель пользователя: **ставить цвет событию на телефоне в Google и видеть этот цвет
на Mac**, а к цвету привязывать индивидуальную кастомизацию.

**Реальность синхронизации:**
- **Цвет календаря** (весь календарь = один цвет) — синкается через CalDAV /
  «Учётные записи интернета», доступен в EventKit (`event.calendar.cgColor`).
  Это то, что мы читаем сейчас.
- **Индивидуальный цвет события** (Google `colorId`, выбранный конкретной
  встрече) — проприетарное расширение Google, **через CalDAV не передаётся**,
  EventKit его не отдаёт. Apple Calendar показывает такое событие цветом
  календаря. → per-event цвет доступен только через **Google Calendar API**.

**Архитектурное решение — «цвет-как-ключ» со сменным источником:**
- В CalendarCore вводим `ColorKey` (нормализованный цвет события) и протокол
  источника цвета. MVP-реализация берёт цвет из EventKit (= цвет календаря).
- Все правила кастомизации (уведомления / строки тикера / скины оверлеев)
  ключуются по `ColorKey`, а не по конкретному источнику.
- Позже (см. поздний этап) добавляем read-only Google API, который подменяет
  источник цвета на per-event `colorId` — **правила кастомизации не переписываются**.

Так «мечта телефон→Mac» не блокирует MVP, но закладывается чисто.

## Поверхности (спецификация MVP)

### Список событий в поповере (menu bar)
- Клик по иконке → окно-поповер (`.menuBarExtraStyle(.window)`) со списком
  ближайших событий: цветовая метка (`ColorKey`), название, время.
- **Группировка по дням**, all-day плашкой сверху дня.
- Клик по событию → открыть в Calendar.app (deep-link).

### Бегущая строка (menu bar, RunCat-style)
- Бегущий текст ближайших событий **в метке самого `MenuBarExtra`** (один
  элемент строки меню, не отдельный `NSStatusItem`): когда событие близко —
  показывает marquee, иначе иконку. Клик всегда открывает поповер-список.
- **Умное появление**: показывается только за N минут до события (дефолт 15), потом прячется.
- **Кастомизация по цвету (позже):** шаблон строки и lead-время могут задаваться
  правилом на `ColorKey` (напр. «красные» события — за 30 мин и с эмодзи).
- ⏳ **TODO (после Фазы 3, по запросу пользователя):**
  - **Подсветка «крутящихся» событий в поповере** — события, которые сейчас
    показываются в бегущей строке (т.е. попали в lead-окно), визуально выделять
    в списке поповера (напр. акцентный фон/значок ⏰), чтобы строка и список были
    связаны. Источник истины — те же `TickerLogic.imminentEntries`.
  - **Прокрутка нескольких событий в строке** — сейчас marquee показывает только
    ближайшее. Продумать показ всех событий из lead-окна: одной непрерывной
    лентой (склейка строк через разделитель) либо поочерёдной сменой каждые
    несколько секунд. Учесть ширину в menu bar и читаемость.

### Оверлеи со скинами
- Прозрачное полноэкранное `NSWindow`, click-through в пустых зонах.
- Триггер по EKAlarm.
- **Движок тем**: рендер по манифесту скина (JSON: ассеты, зоны текста, анимация, длительность, звук, расположение кнопок).
- **Формат темы** `.subnotes-theme` (папка + manifest.json). **Встроенные + пользовательские** темы (кладёшь папку, без пересборки).
- MVP: **2 встроенных скина** — минималистичная карточка + анимированный «самолёт» (SpriteKit) как демо.
- Кнопки плавающим слоем поверх любого скина: **Отложить / Закрыть / Открыть / Подключиться** (Meet/Zoom — парсим из события).
- **Кастомизация по цвету (позже):** выбор скина/звука/длительности может
  задаваться правилом на `ColorKey`.

### Настройки
Какие календари участвуют, горизонт дней, N минут для ticker, snooze-интервалы,

## Ведение issues и автономная выходная рутина

Работа по фазам трекается **GitHub issues**, нарезанными строго из этого плана
(без раздувания — никаких фич мимо PLAN). По выходным репозиторий обходит
**автономная рутина** (remote Claude-агент, сб+вс), поэтому метки — это контракт,
по которому она решает, за что браться.

**Метки issue:**
- `phase-N` — к какой фазе относится. Рутина берёт `core`-issue по возрастанию N.
- `core` — доводится до зелёного **CI** без GUI (логика CalendarCore: парсеры,
  модели, планировщики + юнит-тесты). Это то, что рутина может закрыть сама.
- `needs-ui` — требует **визуальной приёмки человеком** (оверлеи, скины, тикер,
  настройки). Рутина такие issue **не трогает** — они остаются пользователю.

**Алгоритм одного прогона рутины:**
1. Есть открытый PR? CI красный → починить его ветку; зелёный/идёт → спать
   (backpressure: пока PR не смержен человеком, новую работу не начинаем).
2. Нет открытых PR → взять приоритетный `core`-issue, реализовать на ветке
   `auto/<issue>-<slug>`, открыть **draft-PR** (`Closes #N`), дождаться зелёного CI.
3. Нет `core`-issue → draft-PR с предложениями развития в рамках плана.

Жёстко: только ветки + draft-PR; рутина не коммитит в `main`, не мержит, не
релизит. **Мерж draft-PR — ручное действие пользователя** и сигнал «принято»,
который разблокирует следующий issue.

> Среда рутины — Linux без Xcode/EventKit, поэтому она проверяет работу только
> через GitHub Actions CI (macOS-раннер), не локальной сборкой.

**Актуализация схемы.** При любой работе, меняющей статус фазы (закрыт issue,
завершена фаза, сдвинут порядок), в том же PR обновлять **схематичный план до
релиза в [README.md](README.md)** (mermaid-диаграмма + таблица статусов): помечать
фазы по мере завершения и держать диаграмму синхронной с разделом «Порядок
реализации» этого плана. Схема не должна отставать от факта.

## Риски

1. **App Group без подписи** — главный, проверяем в Фазе 0.
2. **Оверлеи поверх fullscreen** — нужен правильный `collectionBehavior`, проверить эмпирически.
3. **Liquid Glass API** — относительно новые материалы, проверить доступность/поведение.
4. **Click-through на анимированных скинах** — кнопки должны оставаться кликабельными поверх прозрачных зон.
